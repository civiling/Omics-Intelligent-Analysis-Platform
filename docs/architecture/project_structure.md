# Project Structure Classification

This file records the target directory layout for the platform after the first backend-oriented reclassification.

## Current Runtime Layout

Core implementation packages now live under `backend/`:

- `backend/agents/`: controlled agent orchestration, routing, planning, specialist agents, and CLI entry points.
- `backend/skill_registry/`: skill definitions, metadata, schemas, method notes, risk notes, and registry validation.
- `backend/workflows/`: workflow configs, workflow runner, executors, placeholder scripts, and run manifests.
- `backend/visualization/`: chart specs, chart validators, result readers, and renderers.
- `backend/reports/`: report collection, section building, review, and Markdown export.

The root-level `agents/`, `skill_registry/`, `workflows/`, `visualization/`, and `reports/` directories are compatibility wrappers. They keep existing imports and commands working while future code can migrate toward `backend.*` imports.

These directories remain at the repository root:

- `tests/`: current automated tests.
- `runs/`: local runtime outputs generated by demo executions.

## Target Backend Layout

- `backend/api/`: future FastAPI layer.
- `backend/api/routes/`: HTTP route modules grouped by feature, such as projects, datasets, workflows, reports, and rag.
- `backend/api/schemas/`: request and response models for API boundaries.
- `backend/services/`: application services that coordinate agents, workflows, reports, storage, and RAG.
- `backend/rag/`: future RAG and GraphRAG layer.
- `backend/rag/ingestion/`: document parsing, chunking, metadata extraction, and indexing jobs.
- `backend/rag/vector/`: vector store integration, embeddings, and semantic retrieval.
- `backend/rag/graph/`: graph database integration, entity/relation models, and GraphRAG traversal.
- `backend/rag/retrievers/`: hybrid retrievers that combine vector, keyword, and graph evidence.
- `backend/storage/`: persistence and file storage adapters.
- `backend/storage/database/`: database connection, migrations, and ORM models.
- `backend/storage/repositories/`: repository classes for projects, datasets, runs, reports, and audit records.
- `backend/storage/object_store/`: local, S3, MinIO, or other artifact storage adapters.
- `backend/tasks/`: background jobs and task queue entry points.
- `backend/config/`: environment-specific settings and runtime configuration.

## Target Frontend Layout

- `frontend/src/api/`: frontend API client wrappers.
- `frontend/src/components/`: shared UI components.
- `frontend/src/features/`: feature modules such as project workspace, workflow runs, reports, and knowledge base.
- `frontend/src/pages/`: route-level pages.
- `frontend/src/stores/`: client-side state stores.
- `frontend/public/`: static assets.

## Infrastructure And Data

- `infra/docker/`: Dockerfiles and compose files.
- `infra/scripts/`: setup, validation, demo, and deployment helper scripts.
- `infra/deployments/`: deployment manifests for server, cloud, or cluster environments.
- `data/examples/`: small demo input datasets.
- `data/raw/`: local raw input data, usually ignored or stored outside Git.
- `data/processed/`: local processed intermediate data, usually ignored or stored outside Git.
- `artifacts/reports/`: exported report files.
- `artifacts/figures/`: exported figures and chart assets.
- `artifacts/exports/`: downloadable result bundles.

## Suggested Migration Order

1. Keep root compatibility wrappers working until downstream scripts and tests migrate.
2. Add `backend/api` and `backend/services` first, importing implementation packages from `backend/`.
3. Add storage models for projects, datasets, runs, reports, and audit traces.
4. Add RAG over existing skill docs, method notes, risk notes, reports, and external literature.
5. Remove compatibility wrappers only after import paths, tests, packaging, and CLI entry points are updated together.
