# Workflow Runtime 工作流模块设计文档

## 1. 设计目标

Workflow Runtime 是平台的任务执行中枢，负责将用户上传数据转化为可运行的分析任务，并对任务进行排队、执行、监控、失败重试、日志记录、结果归档和报告生成。

平台不能由前端直接调用脚本，也不能让每个分析模块独立处理任务状态。所有分析流程应统一纳入 Workflow Runtime 管理。

---

## 2. 总体架构

推荐架构：

前端页面  
→ 后端 API  
→ 任务队列  
→ Workflow Runtime  
→ Worker  
→ Docker / Conda / Singularity 分析环境  
→ 对象存储 / 文件系统  
→ 结果解析  
→ 可视化生成  
→ 报告生成

---

## 3. 核心组成

Workflow Runtime 由以下部分组成：

1. Workflow Definition：工作流定义。
2. Task Definition：任务节点定义。
3. Workflow Scheduler：工作流调度器。
4. Task Queue：任务队列。
5. Worker：任务执行器。
6. Runtime Environment：运行环境。
7. Object Registry：输入输出对象注册。
8. Log Manager：日志管理。
9. Result Parser：结果解析。
10. Failure Handler：错误处理与重试。
11. Report Trigger：报告触发器。

---

## 4. 任务状态

每个任务应支持以下状态：

| 状态 | 含义 |
|---|---|
| pending | 已创建，尚未进入队列 |
| queued | 已进入队列，等待执行 |
| running | 正在运行 |
| failed | 运行失败 |
| completed | 运行完成 |
| cancelled | 用户取消 |
| skipped | 因条件不满足被跳过 |

每个工作流应支持以下状态：

| 状态 | 含义 |
|---|---|
| initialized | 工作流已初始化 |
| running | 工作流运行中 |
| partially_failed | 部分任务失败 |
| failed | 工作流失败 |
| completed | 工作流完成 |
| cancelled | 工作流取消 |

## 7. 工作流一：数据接入与格式识别工作流

### 7.1 适用场景

用户上传新数据后自动运行。

### 7.2 输入

1. UploadedFile
2. 可选 metadata.csv

### 7.3 输出

1. ExpressionMatrix
2. SampleMetadata
3. 数据可分析性摘要
4. 格式识别报告

### 7.4 任务节点

1. FileValidation
2. CompressionCheck
3. MatrixFormatDetection
4. MatrixOrientationDetection
5. RawCountDetection
6. GeneIdTypeDetection
7. SpeciesDetection
8. MetadataDetection
9. MatrixStandardization
10. DataReadinessReport

### 7.5 关键规则

1. 如果矩阵方向为 cell × gene，应提示或自动转置。
2. 如果数值不是整数 raw count，应标记为 normalized matrix。
3. 如果没有 metadata，应生成 metadata 模板。
4. 如果存在 GSM 编号，可尝试解析公开数据库信息生成草稿 metadata。
5. 如果格式无法识别，应停止后续分析并给出错误原因。

## 10. 工作流四：分析模式自动判定工作流

### 10.1 适用场景

数据接入和 metadata 检查完成后自动运行。

### 10.2 输入

1. ExpressionMatrix
2. SampleMetadata
3. QCResult
4. CellTypeAnnotation 可选

### 10.3 输出

1. AnalysisModeRecommendation
2. ConfidenceGateResult
3. 用户提示信息

### 10.4 任务节点

1. SampleCountCheck
2. ConditionCheck
3. GroupCountCheck
4. ReplicateCountCheck
5. PairingCheck
6. BatchCheck
7. CellTypeAvailabilityCheck
8. CellCountPerCellTypeCheck
9. AnalysisModeDecision
10. ConfidenceLevelAssignment

### 10.5 判定规则

1. 只有一个样本：单样本细胞组成分析。
2. 多个样本但无 condition：多样本整合分析。
3. 有 condition 但只有一组：同组整合分析。
4. 有两组但每组样本数不足：探索性 pseudobulk 差异分析。
5. 每组有足够生物学重复：正式 pseudobulk 差异分析。
6. 有 patient_id 且配对完整：配对 pseudobulk 差异分析。
7. 有三个及以上 condition：多组比较或趋势分析。
8. 有 time_point：时间序列分析。
9. 有 dose：剂量响应分析。

## 11. 工作流五：pseudobulk 差异分析工作流

### 11.1 适用场景

用户有多个样本、明确 condition，并完成细胞类型注释。

### 11.2 输入

1. Raw ExpressionMatrix
2. SampleMetadata
3. CellTypeAnnotation

### 11.3 输出

1. PseudobulkMatrix
2. DEGResult
3. RankedGeneList
4. VisualizationResult
5. EnrichmentResult 可选
6. ReportObject

### 11.4 任务节点

1. CellTypeSplit
2. SampleCellTypeCountAggregation
3. LowCellCountGroupFiltering
4. PseudobulkMatrixGeneration
5. DesignFormulaGeneration
6. DifferentialExpressionAnalysis
7. DEGTableFormatting
8. RankedGeneListGeneration
9. VolcanoPlotGeneration
10. MAPlotGeneration
11. DEGHeatmapGeneration
12. DEGResultQualityAssessment

### 11.5 统计模型

非配对设计：

design = ~ condition

有 batch：

design = ~ batch + condition

配对设计：

design = ~ patient_id + condition

多因素设计：

design = ~ batch + patient_id + condition

### 11.6 分析工具

第一阶段建议支持：

1. DESeq2
2. edgeR

后续可支持：

1. limma-voom
2. muscat
3. dreamlet

### 11.7 关键限制

1. 不得把单个细胞作为生物学重复。
2. 正式差异分析必须基于 sample-level pseudobulk。
3. 如果每组样本数不足，应降级为探索性分析。
4. 富集分析继承 DEG 结果的可信度等级。

## 16. 错误处理机制

### 16.1 常见错误

1. 文件格式无法识别。
2. 矩阵为空。
3. 基因数过少。
4. 细胞数过少。
5. 缺少 metadata。
6. condition 不足两组。
7. 每组样本数不足。
8. cell type 中细胞数过低。
9. DESeq2 模型不可拟合。
10. 数据库缺失。
11. 容器运行失败。
12. 内存不足。
13. 任务超时。

### 16.2 错误处理原则

1. 能自动修复的问题尽量自动修复，并记录处理方式。
2. 不能自动修复的问题应给出明确原因。
3. 不应输出虚假的正式结果。
4. 不满足条件时应自动降级分析模式。
5. 失败任务应允许重试。
6. 所有错误必须写入日志。

4. SCENIC 工作流。
5. 空间转录组工作流。
