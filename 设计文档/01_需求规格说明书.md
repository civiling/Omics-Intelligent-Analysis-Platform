# 单细胞智能分析平台需求规格说明书

## 1. 项目定位

本平台定位为面向科研用户的单细胞 RNA-seq 智能分析平台。平台目标不是简单封装 Seurat、Scanpy、DESeq2、CellChat、SCENIC 等工具，而是围绕“数据能否分析、适合做什么分析、结果是否可靠、如何解释结果”构建完整分析闭环。

平台核心流程为：

用户上传数据  
→ 数据格式识别  
→ Metadata 检查与实验设计识别  
→ 自动判定可执行分析模式  
→ 运行标准化分析工作流  
→ 输出可视化结果  
→ 生成带可信度标注的分析报告

平台第一阶段重点支持单细胞 RNA-seq 表达矩阵分析，后续扩展至 bulk RNA-seq、空间转录组、scATAC-seq、CITE-seq、TCR/BCR、微生物组和代谢组等多组学方向。

---

## 2. 建设目标

### 2.1 总体目标

建设一个能够根据用户上传数据自动判断分析可行性、自动选择合适分析模式、执行标准化流程、输出可视化结果和科研报告的单细胞智能分析平台。

### 2.2 第一阶段目标

第一阶段实现单细胞 RNA-seq 基础分析闭环：

1. 支持用户上传常见单细胞表达矩阵。
2. 自动识别数据格式、矩阵方向、数值类型和物种。
3. 支持 metadata 模板生成与实验设计识别。
4. 支持单样本细胞组成分析。
5. 支持多样本整合分析。
6. 支持探索性或正式 pseudobulk 差异分析。
7. 支持 GO、KEGG、Reactome、GSEA 等富集与通路分析。
8. 支持细胞类型自动注释与人工复核。
9. 支持结果可信度评分。
10. 支持自动生成标准科研报告。

## 3. 用户角色

### 3.1 普通科研用户

主要需求：

1. 上传单细胞表达矩阵。
2. 查看数据质量。
3. 获得细胞聚类和细胞类型注释。
4. 查看细胞组成。
5. 比较 normal vs tumor 或 control vs treatment。
6. 查看差异基因和富集通路。
7. 导出图表和报告。

### 3.2 生信分析人员

主要需求：

1. 检查数据格式与 metadata 是否完整。
2. 选择分析参数。
3. 查看分析日志。
4. 下载中间结果和最终结果。
5. 复核细胞注释。
6. 调整分析设计。
7. 重新运行部分工作流。

### 3.3 平台开发人员

主要需求：

1. 扩展分析模块。
2. 管理标准数据对象。
3. 管理工作流节点。
4. 管理数据库版本。
5. 维护任务执行系统。
6. 处理错误日志和任务失败问题。

## 4. 功能范围

### 4.1 第一阶段纳入范围

第一阶段建设以下模块：

1. 数据接入与格式识别子系统。
2. Metadata 构建与实验设计识别子系统。
3. 分析模式自动判定子系统。
4. 结果可信度评分与质控门控子系统。
5. 标准数据对象管理子系统。
6. Workflow Runtime 工作流执行子系统。
7. 参考数据库与知识库子系统。
8. 细胞类型自动注释与人工复核子系统。
9. 前端可视化子系统。
10. 自动报告与解释生成子系统。
11. 文献证据与 RAG 解释子系统的基础版本。

### 4.2 第一阶段暂不纳入范围

第一阶段暂不建设：

1. 项目管理、权限和审计子系统。
2. 多用户团队协作。
3. 企业级数据权限隔离。
4. 空间转录组分析。
5. scATAC-seq 分析。
6. CITE-seq 分析。
7. TCR/BCR 克隆分析。
8. 微生物组和代谢组整合分析。
9. 完整商业化计费系统。
10. 临床诊断或医疗决策功能。

## 5. 数据接入需求

### 5.1 支持格式

第一阶段应支持：

1. csv
2. csv.gz
3. tsv
4. tsv.gz
5. 10x mtx
6. 10x h5
7. h5ad
8. rds
9. loom
10. metadata.csv

### 5.2 自动识别内容

平台应自动识别：

1. 矩阵方向：gene × cell 或 cell × gene。
2. 数值类型：raw count、normalized expression、scaled expression。
3. 基因命名：Gene Symbol 或 Ensembl ID。
4. 物种：human、mouse 或 unknown。
5. 细胞 barcode 格式。
6. 样本数量。
7. 基因数量。
8. 细胞数量。
9. 是否存在重复基因名。
10. 是否存在空行或空列。
11. 是否为压缩文件。
12. 是否有 metadata 文件。

### 5.3 数据接入输出

数据接入完成后，应输出数据可分析性摘要：

1. 文件识别结果。
2. 表达矩阵维度。
3. 是否为 raw count。
4. 是否适合 QC。
5. 是否适合聚类。
6. 是否适合差异分析。
7. 是否需要补充 metadata。
8. 发现的格式问题。
9. 推荐的下一步分析模式。

## 6. Metadata 与实验设计需求

### 6.1 Metadata 最小字段

正式分析至少应支持以下字段：

1. sample_id
2. file_name
3. condition
4. patient_id
5. batch
6. tissue
7. species
8. disease
9. time_point
10. dose
11. paired_status

其中，正式差异分析最关键字段为：

1. sample_id
2. condition
3. patient_id
4. batch

### 6.2 Metadata 构建方式

平台应支持：

1. 用户上传 metadata.csv。
2. 平台自动生成 metadata 模板。
3. 从文件名解析 sample_id。
4. 从 GSM 编号解析 GEO 信息。
5. 自动识别 normal、tumor、control、treatment 等关键词。
6. 自动识别配对样本。
7. 用户手动修改 metadata。
8. 分析前确认 metadata。

### 6.3 实验设计识别

平台应自动识别以下实验设计：

1. 单样本分析。
2. 多样本无分组分析。
3. normal vs tumor。
4. control vs treatment。
5. 配对 normal/tumor。
6. 多组比较。
7. 时间序列。
8. 剂量梯度。
9. 疾病分期。
10. 治疗前后比较。
11. 批次设计。

---

## 7. 分析模式自动判定需求

### 7.1 判定逻辑

平台应根据以下信息自动推荐分析模式：

1. 样本数量。
2. 是否有 metadata。
3. 是否有 condition。
4. condition 是否至少包含两组。
5. 每组样本数量。
6. 是否有 patient_id。
7. 是否为配对设计。
8. 是否有 batch。
9. 是否已有细胞类型注释。
10. 是否为肿瘤数据。
11. 是否有时间点。
12. 是否有剂量信息。

### 7.2 支持模式

平台应支持以下分析模式：

| 数据条件 | 推荐分析模式 | 结果等级 |
|---|---|---|
| 1 个样本 | 单样本细胞组成分析 | 描述性结果 |
| 多个样本，无 condition | 多样本整合与细胞注释 | 描述性结果 |
| 有 condition，但只有 1 组 | 同组整合分析 | 描述性结果 |
| 有 Normal/Tumor，但重复不足 | 探索性 pseudobulk 差异分析 | 探索性结果 |
| 每组有多个生物学重复 | 正式 pseudobulk 差异分析 | 正式统计结果 |
| 有 patient_id 且成对 | 配对 pseudobulk 差异分析 | 正式统计结果 |
| condition ≥ 3 | 多组比较或趋势分析 | 视重复数决定 |
| 有 time_point | 时间序列分析 | 视实验设计决定 |
| 有 dose | 剂量响应分析 | 视实验设计决定 |
| 已有细胞类型注释 | 细胞比例差异、细胞通讯 | 推断性结果 |
| 肿瘤数据 | 肿瘤微环境、CNV 推断 | 推断性结果 |

## 8. 结果可信度评分需求

平台应将结果分为四类：

### 8.1 描述性结果

适用于：

1. 单样本分析。
2. 多样本但无分组。
3. cluster marker 分析。
4. cell type marker 分析。
5. 细胞组成描述。

### 8.2 探索性结果

适用于：

1. 有分组但样本重复不足。
2. 配对信息不完整。
3. 每个 cell type 中样本数不足。
4. 用户希望进行初步候选基因筛选。

### 8.3 正式统计结果

适用于：

1. metadata 完整。
2. condition 明确。
3. 每组有足够生物学重复。
4. 使用 sample-level pseudobulk。
5. 模型中考虑 batch 或 patient_id。

### 8.4 推断性结果

适用于：

1. 细胞通讯分析。
2. 轨迹分析。
3. 转录因子调控分析。
4. CNV 推断。
5. ligand-target 推断。

8. tSNE。
9. 聚类。
10. marker gene 识别。
11. 细胞类型注释。

13. marker 可视化。

## 10. 差异分析需求

平台应支持：

1. cluster marker 差异分析。
2. cell type marker 差异分析。
3. normal vs tumor 差异分析。
4. control vs treatment 差异分析。
5. 配对差异分析。
6. 多组比较。
7. cell-type-specific pseudobulk 差异分析。
8. DEG 上调和下调基因表。
9. 火山图。
10. MA 图。
11. 差异基因热图。
12. DEG 结果下载。

正式差异分析必须基于 sample-level pseudobulk，避免将单个细胞误作为生物学重复。

---

## 11. 富集与通路分析需求

平台应支持：

1. marker gene GO 富集。
2. marker gene KEGG 富集。
3. marker gene Reactome 富集。
4. DEG 上调基因富集。
5. DEG 下调基因富集。
6. ranked gene list GSEA。
7. Hallmark gene set 分析。
8. 用户自定义 gene set。
9. module score。
10. pathway score 可视化。

富集分析的可信度不得高于上游差异分析结果的可信度。

5. negative_markers
6. reference_database
7. annotation_method

## 15. 文献证据与 RAG 解释需求

RAG 子系统第一阶段用于辅助解释分析结果，不直接替代统计结论。

系统应区分：

1. 数据中观察到的结果。
2. 文献或数据库已有证据。
3. 平台基于数据和文献作出的解释。

RAG 子系统支持：

1. marker gene 文献解释。
2. 细胞类型功能解释。
3. DEG 生物学解释。
4. GO/KEGG/Reactome 通路解释。
5. 肿瘤微环境解释。
6. 候选基因功能总结。
7. 后续实验建议。

## 17. 非功能需求

### 17.1 可扩展性

平台应支持后续接入：

1. 空间转录组。
2. scATAC-seq。
3. CITE-seq。
4. TCR/BCR。
5. bulk RNA-seq。
6. 微生物组。
7. 代谢组。
8. 多组学整合。

### 17.2 可追溯性

每次分析应记录：

1. 输入文件。
2. 输入参数。
3. 分析模式。
4. 软件版本。
5. 数据库版本。
6. 开始时间。
7. 结束时间。
8. 运行日志。
9. 输出文件。
10. 错误信息。

### 17.3 可解释性

每个重要结果应附带：

1. 图表。
2. 表格。
3. 分析方法。
4. 结果等级。
5. 可信度说明。
6. 局限性提示。
7. 文献或数据库证据。

## 18. 第一阶段验收标准

第一阶段完成后，应满足：

1. 用户可上传单细胞表达矩阵。
2. 平台可自动识别矩阵格式。
3. 平台可提示是否缺少 metadata。
4. 平台可生成 metadata 模板。
5. 平台可完成 QC、聚类、UMAP、marker gene。
6. 平台可进行细胞类型自动注释。
7. 用户可人工修改细胞类型注释。
8. 平台可进行细胞组成分析。
9. 平台可根据 metadata 判断能否做差异分析。
10. 平台可进行 pseudobulk 差异分析。
11. 平台可进行 GO/KEGG/GSEA 富集。
12. 平台可生成主要图表。
13. 平台可生成标准科研报告。
14. 报告中包含结果可信度标注。
15. 报告中包含方法、参数、软件版本和数据库版本。
